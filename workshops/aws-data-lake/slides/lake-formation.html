Section 1 of 3 — Introduction

AWS Lake Formation

Introduction, Grant Models & Demo Lessons

A service for centrally governing, securing, and globally sharing data used for analytics and ML.

  • Manages fine-grained access control on data lake data in S3 and metadata in the AWS Glue Data Catalog
  • Provides its own permissions model that augments IAM — doesn't replace it
  • Grant/revoke mechanism similar to a relational database (RDBMS)
  • Consistent enforcement across AWS analytics and ML services
  • Supports sharing data across multiple AWS accounts, AWS Organizations, or directly with IAM principals

Permissions apply to

  • IAM users and roles
  • Users / groups federated through an external IdP

Fine-grained controls cover

  • Database and table level
  • Column level
  • Row level (data filters)
  • Cell level (row + column filter combined — not a separate mechanism)

Where Lake Formation Enforces Permissions

Permissions apply to IAM users and roles, and to users/groups federated through an external IdP. Enforcement is consistent across all integrated AWS analytics and ML services.

Integrated AWS services

AWS Service LF Permissions Enforced
Amazon Athena ✅
Amazon QuickSight ✅
Amazon Redshift Spectrum ✅
Amazon EMR (Apache Spark) ✅
AWS Glue ETL ✅

Enforcement is consistent regardless of service

Whether a user queries via Athena, runs a Glue job, or reads from QuickSight — the same Lake Formation permission rules apply. The access control lives in one place, not embedded in each service.

Key insight: Before Lake Formation, an analyst with Athena access and a data engineer with Glue access might both read columns they shouldn't — because each service enforced only IAM S3 bucket permissions. LF adds a layer that applies uniformly.

Introduction

Feature Areas at a Glance

Lake Formation organizes its capabilities into three pillars, each solving a distinct data governance concern.

Data Ingestion & Management

Bringing data into the data lake, cataloging schemas, labeling resources with tags for access control, and scheduling incremental loads from database sources.

Security Management

Defining and enforcing access controls at database, table, column, row, and cell levels. Includes audit logging, cross-account sharing, ABAC, and hybrid access mode.

Bring Data Into the Catalog (Federation)

Connecting external data sources and Redshift catalogs without migrating metadata. Includes Redshift integration, external metastores, S3 Tables, and AWS Data Exchange.

These pillars overlap in practice — you ingest data into the catalog, then apply security rules, then optionally federate access to it from external systems. All three are centrally managed from the Lake Formation console.

Ingestion & Security Capabilities

Bringing data in

  • Import from RDS/EC2-hosted MySQL, PostgreSQL, SQL Server, MariaDB, Oracle, or on-prem sources via JDBC — bulk or incremental
  • Glue Crawlers auto-discover schema from S3 into the searchable Data Catalog

Crawlers + LF-Tags together are the foundation of a self-healing data lake: new data lands on S3, a crawler discovers its schema, tags are applied via automation, and a security rule written once auto-applies to new assets.

Introduction

Hybrid Access Mode

  • Lets you secure and access cataloged data using both LF permissions AND IAM permissions policies (for S3 and Glue actions)
  • Onboard LF permissions selectively and incrementally — one use case at a time, not a big-bang migration
  • Enables LF permissions for specific users without interrupting existing IAM-only workloads
The mechanic that matters everywhere in this module: once LF is enabled on a resource, both layers must Allow — IAM governs whether the principal can call the AWS API at all, LF governs which data the query can actually see. A Deny in either layer blocks access; neither layer alone is sufficient.

BEFORE — IAM-only

🛡
Single IAM Shield
IAM policies control all access to all S3 + Glue resources

AFTER — Hybrid Mode

🔶
LF Layer (enforced)
New resources — column/row/cell security via LF grants
🛡
IAM Layer (still active)
Existing workloads — unchanged IAM policies
Before you enable LF: designate an LF admin, register S3 paths as data lake locations, confirm the LF service role can reach S3.

Grant Models

Named Data Catalog resources vs. resources matched by LF-Tags — two fundamentally different mental models behind the same Grant button.

Grant Models

Two Ways to Grant in Lake Formation

When you click Grant in the LF console, you must pick a model — same button, two different mental models.

Property Named Data Catalog Resources Resources Matched by LF-Tags
How you specify it Pick a database, table, optionally specific columns — by exact name Write a tag expression, e.g. sensitivity=public AND department=finance
Scope Static — exactly the resources you ticked Dynamic — every resource matching the predicate, today and in the future
New tables tomorrow Not covered — need a new grant per resource Automatically covered once tagged — zero new grants
Audit "what can this user read?" Easy — Data Permissions page shows db → table directly Harder — must join the tag expression with current tag attachments
Best for Small lakes, one-off grants, ad-hoc sharing Large orgs, many tables, a consistent classification policy

SQL analogue — Named

GRANT SELECT
  ON quicklabs_student8_lake.curated_oil
  TO student7;

SQL analogue — LF-Tags (conceptual — standard SQL has no equivalent)

GRANT SELECT
  ON ANY TABLE
  WHERE classification = 'public'
  TO student7;
Illustrative example of where named grants get painful: a data lake with 200 tables and 15 analysts needing different column-level access to each table implies on the order of 3,000 individual named grants — and every new table adds more. LF-Tags collapse this to one rule per policy plus a tag attachment per resource.

Grant Models

When the Difference Matters

Scenario Named grants LF-Tag grants
1 table, 1 consumer 1 grant 1 grant + 1 tag attachment
50 tables, 1 consumer, same access 50 grants 1 grant + 50 tag attachments
Add a 51st table next week +1 more grant manually Just tag the new table — grant auto-applies
Audit "what can student7 read?" Look at Data Permissions list — direct Look at tag expression + every table's current tags
Coordination cost Every new table = ticket to security team Data eng tags; security wrote the rule once
The architectural punchline: named grants couple access control to resource creation. Tag-based grants decouple them. This is the difference between "security is a bottleneck" and "security scales" — at 500+ tables, named grants become operationally untenable.

Demo Lessons

How LF Actually Intercepts a Query

What you're controlling IAM LF
"Can this user use Athena at all?" ✅ —
"Can this user see column volume?" — ✅
"Can this user see rows where month = 12?" — ✅
"Can this user read S3 directly?" Required, but empty LF issues temp creds at query time
Punchline: IAM stays static per role. LF grants are where access changes. Three users with identical IAM policies can see three different result sets from the same query.

The five-step flow

1
Athena receives the query and asks Glue Catalog for the table's location + asks LF "does this principal have SELECT?"
2
LF evaluates every applicable grant — named grants, LF-Tag expressions, and data cell filters, combined
3
LF returns a GetDataAccess response: a short-lived credential scoped to allowed columns, allowed rows, and the registered S3 prefix only
4
Athena reads S3 using those temp creds — never the user's own IAM identity
5
Athena returns already-filtered rows; every read is a GetDataAccess event in CloudTrail — the user cannot bypass LF via direct S3 access, since they never hold S3 credentials

Demo Lessons

Same Query, Three Views — and How to Prove It

SELECT * FROM quicklabs_student8_lake.curated_oil LIMIT 5;
Persona Result
Data Lake Admin All 11 columns, all 6,367 rows
Analyst (column grant — no volume) 10 columns (volume gone), all 6,367 rows
Analyst with data filter (month = 12) 10 columns (volume gone), ~530 rows
Tagged analyst (sensitivity = public) 10 columns (volume tagged restricted), all rows
Different grant models, different result sets — all from the same query string. Governance is declarative (grant a rule), not imperative (rewrite the query, build a view, manage S3 prefixes by hand).
Auditing it: every GetDataAccess event records the principal, the table, the exact authorized column list, the Athena query ID, and a timestamp. That's a materially different audit capability than traditional S3 access logs — "Alice read s3://bucket/x.parquet" tells you the object, not which columns were actually returned or authorized.
